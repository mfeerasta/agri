# Data model

All Zameen tables live in the `zameen` schema in the shared Supabase project (co-tenant with Haazri's `public.*`). Cross-schema joins to Haazri are allowed where needed for unified worker IDs.

## Domain map

```
entities
  └─ farms
       └─ blocks
            └─ fields
                 └─ plots
                 └─ soil_tests
                 └─ crop_plans
                      └─ crop_stage_logs
                      └─ harvest_records
                      └─ cost_allocations  (also link directly to fields)

entities
  └─ users (auth.users mirror, also linked to haazri_worker_id)
       └─ user_entity_roles  (RBAC + per-role approval limits + delegation)

entities
  └─ assets
       ├─ asset_hour_meters
       ├─ asset_logs
       ├─ diesel_daily_logs    (per asset per day)
       └─ repair_requests
              ├─ repair_quotes (>=1, one selected)
              ├─ repair_work_orders
              └─ parts_replaced
```

## Approval and feasibility

```
approval_workflows (entity_id, approval_type, version, thresholds_pkr)
approval_requests  (state machine instance; carries payload + context_snapshot)
   └─ approval_actions  (append-only audit; GPS, IP, UA captured)
feasibility_studies
   ├─ feasibility_attachments
   └─ feasibility_comments
```

Every business record that requires approval keeps a nullable `approval_request_id` foreign key (e.g. `diesel_purchases.approval_request_id`). The approval state is read from `approval_requests.state` — never duplicated.

## Cost allocation

`cost_allocations` is the single source of truth for "what did this field/crop cost?" Every cost-bearing module writes here:

```
cost_allocations (
  source_module,  -- 'diesel' | 'repair' | 'input' | 'labor' | ...
  source_record_id,
  field_id, crop_plan_id, asset_id,
  cost_pool,       -- 'seed' | 'fertilizer' | 'diesel' | 'repairs' | ...
  amount_pkr,
  allocated_on,
  allocation_key   -- 'direct' | 'proportional:hours' | etc.
)
```

`computeFieldPnL(cropPlanId)` aggregates by `crop_pool`, joins to harvests and settlements, and produces the per-field, per-crop, per-season P&L.

## RLS strategy

- Tables with `entity_id`: visible iff `entity_id IN accessible_entity_ids(auth.uid())`. Generated by `0002_rls_policies.sql`.
- `users`: self-row or other users in your accessible entities.
- Master data (`crop_profiles`, `permissions`, `market_prices`): readable by any authenticated user.
- `audit_log`: append-only. No update/delete policy ⇒ denied.

## Storage buckets

- `zameen-receipts` — diesel and input receipts. Authenticated R/W within entity.
- `zameen-documents` — generic entity docs.
- `zameen-feasibility` — feasibility study attachments.
- `zameen-worker-docs` — worker CNIC scans etc. Read scoped to self or HR roles.

High-volume geotagged field photos go to Cloudflare R2, not Supabase Storage, because of egress economics.

## Money

- DB columns: `decimal(p, 2)` PKR.
- Application: `bigint` paisa via `@zameen/shared/money` to avoid float drift.
- Display: `<Pkr value={...} mode="plain|lac_crore" />`.
- No multi-currency support. Foreign currency notes stored only in free-text fields for reference.

## Audit walk

Given any GL entry, you can walk:

```
journal_lines.journal_entry_id
  → journal_entries.source_record_id   (e.g. a diesel_purchases.id)
  → diesel_purchases.approval_request_id
  → approval_requests.id
  → approval_actions (full chain of who decided what, with GPS + IP)
```

This is the platform's value proposition to MF and family: every number traces back to the decision and the person.
